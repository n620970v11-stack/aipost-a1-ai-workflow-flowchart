## 第 2 章：任務背景（一）｜關於我與工作痛點 ✅

**關於我**
- 金融業 系統分析師（System Analyst, SA）
- 工作主要職責：需求訪談、流程圖繪製、PRD 撰寫

**繪製流程圖時遇到的痛點（使用 drawio 作為繪圖工具）**
1. 初版流程建構難度高
將訪談內容、需求文件或口頭討論整理為具有邏輯性的第一版流程圖，是整體流程圖製作中最具挑戰且最耗費心力的階段
2. 手動繪製耗時，降低工作效率
流程圖繪製需投入大量時間於節點建立、連線、版面配置及格式調整等重複性操作，壓縮了分析流程與驗證邏輯的時間
3. 複雜流程容易產生品質風險
當流程涉及多部門協作、跨系統整合或多重判斷條件時，手動繪製容易遺漏分支、例外情境或流程關聯，影響流程圖的完整性與正確性
4. 流程調整成本高，維護彈性不足
當需求或流程發生變更時，往往需要同步調整多個節點與連線，修改範圍廣且耗時，亦容易因遺漏而產生流程不一致的情況
5. AI 產圖容易產生幻覺，增加重新檢視成本
AI 產圖或修改流程圖時，可能出現無依據新增、遺漏既有內容，或「改 A 壞 B」的情況。即使已完成產圖，仍需重新逐一核對節點、分支與連線，往往再次投入大量時間檢視，抵銷原先預期節省的工時

## 第 3 章：任務背景（二）｜我的應用方式與解法 ✅

**為什麼選擇 draw.io MCP**

1. 符合現有的團隊協作方式
團隊成員與 BU 單位目前皆使用 draw.io 網頁版繪製及維護流程圖。使用相同工具與檔案格式，方便不同角色直接開啟圖面，在畫布上修改節點、連線與調整版面，有利於後續交付及共同維護

2. Mermaid 不符合本次的圖面調整需求
Mermaid 主要透過語法生成圖形，並由排版演算法自動計算節點位置與連線路徑。當流程包含多角色、多重判斷、迴圈或跨區連線時，產生的版面不一定符合實際閱讀需求；若只想移動單一節點、調整連線路徑或修改局部版面，通常需要修改語法，整體上操作難度較高且不直觀

3. draw.io MCP 提供新的實測機會
官方 draw.io MCP 套件於 2026 年 2 月推出，讓 AI 能將整理後的流程內容轉換成可在 draw.io 中開啟及繼續編輯的圖面。因此，本次希望實測能否在維持既有 draw.io 協作方式的前提下，加入 AI 自動分析與產圖能力，兼顧生成效率與人工修改彈性。

**draw.io MCP 的四種整合方式**

| 整合方式 | 運作方式 |
| :--- | :--- |
| MCP App Server | 在支援 MCP Apps 的 AI 對話介面中直接預覽流程圖，並可再開啟至 draw.io 編輯 |
| MCP Tool Server | 以 MCP Server 連接 Claude Code 等 MCP Client，將產生的流程圖直接開啟於 draw.io 網頁版 |
| Skill + CLI | 透過 Claude Code Skill 與 draw.io Desktop CLI 產生 `.drawio` 檔案，需使用 draw.io 桌面版 |
| Project Instructions | 將 draw.io 繪圖規則加入 Claude Project，由 Claude 產生可開啟至 draw.io 的連結，不需安裝 MCP Server |

**本次選擇 MCP Tool Server 的原因**

1. 符合 IDE 操作習慣
本次主要在 Claude Code 的 IDE 環境中進行實測，因此未選擇以 Claude Project 為主的 Project Instructions。

2. 符合 draw.io 網頁版使用習慣
MCP Tool Server 可將 AI 產生的流程圖直接開啟於 draw.io 網頁版，符合目前團隊使用瀏覽器開啟及編輯圖面的方式；相較之下，Skill + CLI 需要搭配 draw.io 桌面版，與本次使用情境不符。

3. 可與自訂 Skill 組成同一套工作流程
自訂 Skill 可統一規範何時呼叫 draw.io MCP、呼叫前需完成哪些流程分析與使用者確認，以及產圖後需要執行哪些檢查步驟。MCP Tool Server 負責提供繪圖工具，Skill 則負責編排整體任務流程，兩者分工明確且可重複使用。

技術參考：[draw.io MCP 官方說明](https://www.drawio.com/docs/manual/generate/drawio-mcp-server/)

**我的應用方式**

繪製流程圖前，我會先提供專案資料（業務場景流程說明文件）作為依據；AI 先理解文件內容，再透過 draw.io MCP 產出第一版流程圖。以真實文件作為上下文，讓 AI 的分析與產圖有明確依據，而不是只依賴單次對話內容或自行推測。

本次設計以一份 Task Prompt 啟動任務並呼叫自訂的 Skill；Skill 再依使用者是否已有流程說明文件，執行對應的流程梳理方式。完成流程內容確認後，再透過 draw.io MCP 產出可編輯的流程圖，最後由人工檢查與調整。使用者仍可直接在 draw.io 畫布上修改節點、連線與版面，保留人工編輯彈性與最終決策權。

**如何解決工作中的痛點**

- 降低初版流程建構難度：AI 先從流程說明文件中整理角色、步驟、判斷條件與例外情境，協助使用者更快建立第一版流程架構
- 減少手動繪製時間：透過 draw.io MCP 自動建立節點、連線與初步排版，減少重複性的繪圖操作，讓使用者將時間投入流程分析與內容確認
- 降低複雜流程的遺漏風險：透過自訂 Skill 規範流程梳理與檢查步驟，協助檢查多角色、多條件分支、確認迴圈與例外情境
- 提高流程調整彈性：需求變更時可先更新流程說明文件，再請 AI 依據新版內容重新分析與生成，減少逐一修改節點與連線的作業
- 降低 AI 產圖幻覺與檢查成本：AI 依流程說明文件產圖，可降低無依據新增、遺漏內容或「改 A 壞 B」的風險；人工也能直接以原始文件作為檢查基準，將全面重查轉為有依據的比對，提升檢查效率

> 以上為本次實測前提出的解法設計與預期效益，實際改善程度將於後續實測中，透過原始輸出、人工修改內容與所需時間進行驗證。

### 第三部分：自訂 Skill 如何運作

自訂 Skill 的資料夾結構如下：

```text
flow-chart-creator/
├── SKILL.md
└── references/
    ├── flow-desc-template.md
    └── drawing-spec.md
```

各檔案功能：

- `SKILL.md`
  - 定義整個任務的執行順序
  - 確認使用者是否已有流程說明文件
  - 規範文件分析、流程梳理、使用者確認、產圖及檢查步驟
  - 決定何時讀取參考文件及何時呼叫 draw.io MCP
- `flow-desc-template.md`
  - 使用者沒有流程說明文件時使用
  - 定義建立流程文件所需的欄位
  - 引導使用者整理角色、系統、子流程、判斷條件、例外情境及輸出結果
- `drawing-spec.md`
  - 定義流程圖的繪圖規格
  - 包含節點樣式、色彩、間距、版面方向、連線方式與可讀性檢查要求

**已有文件與尚無文件的處理流程**

```text
Skill 詢問：是否已有流程說明文件？
        ↓
┌──────────────────┴──────────────────┐
↓                                     ↓
已有文件                              尚無文件
讀取並分析文件                        讀取 flow-desc-template.md
                                      ↓
                                      依模板欄位逐步詢問
                                      並建立流程說明文件
└──────────────────┬──────────────────┘
                   ↓
        選擇要繪製的圖表類型
                   ↓
        選擇繪圖範圍
        （完整主流程／部分子流程）
                   ↓
        依指定範圍整理角色、系統、
        子流程、判斷點與例外情境
                   ↓
        使用者確認流程內容是否正確
                   ↓
        確認無誤後才進入繪圖階段
```

### 第四部分：Claude Fable 5 如何呼叫 draw.io MCP

1. Claude Fable 5 依自訂 Skill 與 `drawing-spec.md`，將使用者確認後的流程內容整理成 draw.io 可使用的 XML。
2. Claude Fable 5 呼叫 draw.io MCP 的繪圖工具，並傳入 XML。
3. draw.io MCP 接收 XML，產生可在 draw.io 網頁版開啟的圖面連結。

本次使用的官方 `@drawio/mcp` 會將 draw.io XML 生成參考規則提供在工具說明中，讓 Claude Fable 5 能依節點、連線、容器及路由等規格產生 draw.io XML。

```text
draw.io MCP 提供工具與 XML 規則
        ↓
Claude Fable 5 依規則生成 XML
        ↓
Claude Fable 5 呼叫 MCP 並送出 XML
```

技術參考：[draw.io MCP Tool Server 官方說明](https://github.com/jgraph/drawio-mcp/blob/main/mcp-tool-server/README.md)

## 第 7 章：實際節省多少時間 🔄

### 比較基準

本次以「完成經人工檢查與必要修改、可供後續討論的第一版子流程圖」作為計時終點，比較人工使用 draw.io 從零構思與繪製，以及使用 AI、自訂 Skill 與 draw.io MCP 的作業時間。

### 人工繪製的預估時間

過去使用 draw.io 人工繪製時，每個子流程從內容構思、節點安排、判斷分支整理到完成第一版圖面，平均約需 60 分鐘。以本次五個子流程計算，預估總工時約為 300 分鐘（約 5 小時），需佔用大半個工作日。

### AI 輔助繪圖的實測時間

AI 工作流不只計算產圖時間，也將前置的流程說明文件撰寫時間納入：

- 撰寫並整理流程說明文件：約 30 分鐘
- AI 生成單一子流程圖：平均約 6 分鐘（從影片推估計算）
- 五個子流程產圖時間：`6 分鐘 × 5 = 30 分鐘`
- 每個子流程的人工檢查及必要修改時間：約 15 分鐘
- 五個子流程的檢查及修改時間：`15 分鐘 × 5 = 75 分鐘`
- AI 工作流總時間：`30 分鐘（文件）+ 30 分鐘（產圖）+ 75 分鐘（檢查及修改）= 135 分鐘`

重試原則：若圖面只有節點位置、連線路徑或文字呈現等局部問題，直接在 draw.io 網頁版進行人工修改，不重新產圖；只有發生流程大幅遺漏、業務邏輯錯誤或 XML 無法正常開啟等重大錯誤時，才重新生成，並將重試時間另行計入。

### 時間效益比較

| 比較項目 | 人工繪製 | AI 輔助繪圖 |
|---|---:|---:|
| 前置構思與文件整理 | 包含於各子流程的繪製時間內 | 約 30 分鐘 |

| 工時降低比例 | — | 約 55% |

### 實測結論

將前置文件整理、AI 產圖及人工檢查修改時間全部納入後，五個子流程的完成時間，預估由人工繪製的 300 分鐘縮短為 135 分鐘，節省約 165 分鐘（2 小時 45 分），整體工時降低約 55%。這表示 AI 工作流的價值並非完全取代人工，而是將原本花在節點建立、連線與初步排版的時間，轉移至更具價值的流程正確性檢查與必要調整；同時，也將原本分散在各子流程中的構思過程，提前沉澱為可重複使用的結構化流程文件。

> 註：本次計算以未發生重大錯誤、不需重新產圖為基準。首次設定 MCP 與問題排除屬於一次性環境建置成本，不納入每次產圖工時，但另於第 10 章揭露。待五個 Demo 完成後，將再依各子流程的實際產圖、檢查修改與重試紀錄更新最終數據。

## 第 8 章：Fable 5 的三項優點與三項限制 🔄

### 三項優點

> 以下為 Fable 5 在本次流程圖任務中展現的優點，不主張這些能力僅限 Fable 5 具備。第 2、3 項待完整主流程實測後定稿。

#### 優點 1：圖表生成與工具呼叫速度快

在流程內容確認完成後，Fable 5 能快速生成 draw.io XML、呼叫 draw.io MCP，並開啟已完成渲染的 draw.io 網頁。目前實測單一子流程從開始生成到 MCP 開啟網頁，約需 6 分鐘，相較於人工逐一建立節點、判斷分支與連線，能夠明顯縮短第一版圖面的產出時間。

> 計時範圍：Fable 5 開始生成 draw.io XML → 呼叫 draw.io MCP → draw.io 網頁開啟。此為端到端產圖執行時間，其中同時包含模型生成、MCP 處理與網頁開啟時間。

#### 優點 2：能正確整合複雜的多子流程邏輯（待驗證）

預計以合併五個子流程的完整主流程進行實測，檢查 Fable 5 是否能在單次任務中，完整保留各子流程的順序、銜接關係、角色、系統、判斷點與例外分支，並確認是否出現內容遺漏、順序錯置或擅自新增流程的情況。

若實測結果正確，可得出以下結論：Fable 5 能在單次任務中整合五個相互銜接的子流程，並維持原始文件中的流程順序、判斷邏輯與跨系統關係，展現處理複雜長流程的內容整合能力。

#### 優點 3：複雜產圖時仍能維持規格遵循與工具執行穩定（待驗證）

同一次完整主流程實測將同時檢查：Fable 5 是否完整遵循自訂 Skill 與繪圖規格、draw.io XML 是否能一次成功解析、MCP 是否能順利開啟 draw.io，以及圖面是否維持節點類型、顏色、系統方塊與連線規則的一致性。同時記錄是否出現重疊、斷線、大幅跑版、MCP 呼叫失敗，以及過程中需要人工糾正或重試的次數。

若實測結果穩定，可得出以下結論：即使產圖範圍由單一子流程擴大至完整主流程，Fable 5 仍能遵循自訂 Skill 與繪圖規格，成功生成可解析的 draw.io XML 並完成 MCP 工具呼叫，過程不需要頻繁人工介入。

### 三項限制

#### 限制 1：單次產圖的 Token 成本偏高，且 XML 輸出會放大費用

Fable 5 的官方 API 定價為每百萬輸入 Token 10 美元、每百萬輸出 Token 50 美元。本次任務除了需讀取流程說明文件、SKILL.md 與繪圖規格，還需輸出篇幅較長的 draw.io XML。當流程節點變多或需要重新產圖時，輸入與輸出 Token 都會繼續累積，尤其高單價的輸出 Token 會直接提高每張圖的成本。

對單次試作而言，成本可能尚可接受；但若將 Fable 5 作為日常繪圖工具，就需要同時評估圖面複雜度、重畫次數與實際 Token 支出。

#### 限制 2：Fable 5 並非無上限包含於訂閱方案，額度用完可能中斷工作

依 Anthropic 2026 年 6 月 30 日公告，Fable 5 於 7 月 7 日後，Pro、Max、Team 與部分 Enterprise 方案需透過 usage credits 繼續使用；標準 Enterprise 席次也沒有內含的 Fable 5 額度。若組織沒有啟用使用點數，或預算已用完，使用者就無法繼續使用 Fable 5。

對需要持續訪談、確認與修改流程的工作而言，額度不只是費用問題，也可能在作業進行中造成中斷。使用者屆時只能申請或購買額外點數、等待組織補充預算，或改用其他模型。

#### 限制 3：BU 多人、多版本反覆迭代時，成本與可用量難以預測

業務流程往往不會只繪製一次。BU 在功能訪談、需求調整、例外情境補充與測試驗證過程中，可能針對同一子流程進行多輪修改。當五個子流程、多次重畫與多位團隊成員的使用量疊加後，原本可接受的單次成本會快速放大。

因此，Fable 5 雖然能夠加速第一版流程圖產出，但若要擴大為 BU 的常態工具，仍需要先建立額度分配、成本上限、模型選用與使用優先級。否則，使用上限與額度預算會成為反覆迭代功能時的實際阻礙。

## 第 9 章：我會推薦給誰、不推薦給誰 🔄

### 推薦判斷的前提

本章的推薦與不推薦對象，是以「Fable 5 搭配本次自訂工作流」為評估前提思考推薦及不推薦的族群。此工作流包含：

```text
Task Prompt
→ 結構化流程說明文件
→ 自訂 flow-chart-creator Skill
→ Fable 5 分析與生成 draw.io XML
→ draw.io MCP 開啟可編輯圖面
→ 人工檢查、修改與定稿
```

### 推薦給誰

#### 推薦給需要快速建立討論初稿的 Senior PM 與 SA

在需求訪談初期，BU 提出的內容通常尚未完全具體，PM 或 SA 需要透過反覆提問，逐步釐清角色、系統、流程步驟、判斷條件與例外情境。此時若直接從空白畫布開始繪製，不僅耗時，也容易因需求持續變動而重複調整。

透過 AI，可以先將訪談內容整理成結構化的流程說明文件，再交由 Fable 5 快速產生第一版流程圖。這張圖不需要在第一次產出時就達到定稿品質，而是作為 PM、SA 與 BU 之間的共同討論基準，讓原本抽象的需求快速轉化為可檢視的視覺內容。

當 BU 看見實際圖面後，通常更容易指出：

- 哪些流程步驟需要補充
- 哪些判斷條件不正確
- 哪些角色或系統被遺漏
- 哪些例外情境需要另外處理

Senior PM 與 SA 本身具備流程分析、圖面判讀及 draw.io 修改能力，因此可以快速檢查 AI 輸出，再以初版圖為基礎進行人工加工與後續迭代。這能將時間從「手動建立第一版圖面」轉移到更有價值的需求判斷與溝通工作。

> 推薦定位：將 Fable 5 作為需求討論初期的「第一版流程圖加速器」，而不是取代 PM、SA 的業務判斷、人工修改與正式定稿責任。

### 不推薦給誰

#### 1. 尚未熟悉流程圖的 Junior PM 與 SA

對剛開始學習需求分析與流程繪製的 Junior PM、SA 而言，當前更重要的是透過反覆練習，理解不同圖表類型的使用時機、各種圖形與節點的意義、判斷分支的畫法，以及如何將業務規則轉換為正確的圖面邏輯。

若在尚未建立這些基礎能力前，就直接依賴 Fable 5 與自動產圖工作流，可能無法辨識節點使用不當、流程遺漏、連線錯誤或業務邏輯不合理等問題。因此，不建議將此工作流作為 Junior PM、SA 學習流程圖時的唯一方式；應先建立人工梳理與繪圖能力，再將 AI 用於加速初稿產出。

#### 2. 需要大量反覆產圖，但沒有足夠額度或預算的團隊

此工作流每次產圖都需要讀取流程文件、Skill 與繪圖規格，並輸出篇幅較長的 draw.io XML。BU 在需求尚未穩定時，又可能因為流程步驟、系統串接、判斷條件或例外情境變動，進行多輪重新產圖。

當子流程數量、修改次數與使用人數增加後，Fable 5 的 Token 費用與 usage credits 消耗也會持續擴大。若團隊沒有建立額度預算、成本上限與模型選用原則，可能在需求迭代過程中因額度不足而無法繼續使用。因此，在高頻產圖情境下，不建議將 Fable 5 作為唯一且預設的產圖模型。

## 第 10 章：失敗案例與問題排除｜draw.io MCP 未在新專案生效 ✅

> 本章以實際發生的問題為案例，說明如何從現象定位根因、完成修正並驗證結果。建議在簡報中拆成兩頁呈現。

### 第 1 頁：發生了什麼問題？

#### 原本預期

流程內容確認後，Fable 5 應呼叫 draw.io MCP 的 `open_drawio_xml` 工具，自動開啟 draw.io 網頁版，並將產生的流程圖渲染至畫布。

#### 實際發生的狀況

第一次產圖時，Claude Code 找不到 `open_drawio_xml` 工具，因此無法按照原定工作流開啟 draw.io 網頁版。當時只能採用替代方案：先將內容儲存為 `.drawio` 檔案，再透過 draw.io 桌面版開啟。

這個現象一開始容易被誤認為 draw.io MCP 安裝失敗，或 Skill 沒有正確執行；但進一步檢查後發現，問題不在 Skill，也不是 MCP 套件本身無法啟動，而是目前專案與 Session 沒有載入該 MCP Server。

#### 問題排除時間軸

| 階段 | 狀態 | 檢查結果與處理方式 |
|---|---|---|
| 第一次產圖 | ❌ 無法開啟網頁版 | 目前專案中沒有可用的 draw.io MCP 設定，`open_drawio_xml` 工具不存在；先改存 `.drawio` 檔並以桌面版開啟。 |
| 檢查既有設定 | 🔍 定位問題 | 雖然曾在其他專案使用 draw.io MCP，但該設定沒有套用至目前專案。 |
| 重新註冊 MCP Server | ✅ 設定完成 | 將官方 `@drawio/mcp` 註冊至 Claude Code，並確認 Server 可以啟動及提供正確的工具清單。 |
| 註冊後的原 Session | ⏳ 工具仍未出現 | 設定雖已完成，但當下 Session 尚未重新建立 MCP 連線，因此仍無法直接呼叫新工具。 |
| 重新啟動 Session | ✅ 呼叫成功 | `open_drawio_xml` 工具成功載入，並以 libavoid 自動繞線開啟 draw.io 網頁版及渲染流程圖。 |

### 第 2 頁：根本原因、修正方式與學習

#### 為什麼其他專案已安裝，這個專案卻不能使用？

「電腦上已有 MCP 套件」與「Claude Code 在目前專案中已註冊並載入 MCP Server」是兩件不同的事。Claude Code 的 MCP 設定具有不同作用範圍：

- `local`：預設範圍，只供目前使用者在目前專案使用
- `project`：透過專案根目錄的 `.mcp.json` 與專案成員共用
- `user`：供同一位使用者跨所有專案使用

因此，若先前是在另一個專案以預設的 `local` 範圍註冊 draw.io MCP，設定只會對該專案生效；切換至本次 A1 專案後，不會自動出現 `open_drawio_xml` 工具。僅確認設定寫入 `~/.claude.json`，也不能直接判斷它是否為跨專案的 `user` 設定，因為 `local` 設定同樣可能依專案路徑儲存在該檔案中。

#### 修正方式

1. 確認官方 `@drawio/mcp` Server 可以正常啟動。
2. 依使用需求，將 draw.io MCP 註冊於目前專案，或設定為跨專案可用的 `user` 範圍。
3. 透過 `claude mcp list`、`claude mcp get drawio` 或 Claude Code 內的 `/mcp`，確認 Server 狀態與工具清單。
4. 設定完成後重新建立 Session，使 Claude Code 載入新的 MCP 連線與工具。
5. 再次呼叫 `open_drawio_xml`，確認 draw.io 網頁版能開啟並正確渲染流程圖。

#### 最終驗證結果

- `open_drawio_xml` 工具已可正常呼叫
- libavoid 自動繞線功能正常運作
- draw.io 網頁編輯器可由 MCP 自動開啟
- 產生的流程圖可直接渲染至畫布並繼續人工編輯

#### 本次學習與後續預防方式

這次問題證明，「Skill 指示 AI 使用工具」不代表該工具已經存在於目前環境中。Skill 負責定義工作流程與呼叫時機；MCP Server 是否已註冊、作用範圍是否正確，以及目前 Session 是否成功載入，仍需要另外確認。

後續在正式 Demo 前，將增加以下環境檢查：

- 確認目前專案能看見 draw.io MCP Server
- 確認 `open_drawio_xml` 已出現在工具清單中
- 先用測試 XML 執行一次網頁開啟與渲染
- 重新啟動 Session 後再開始正式錄影
- 若需跨專案重複使用，確認 MCP 註冊於適當的 `user` 範圍

> 案例結論：這次不是 Skill 撰寫錯誤，也不是 draw.io MCP 套件安裝失敗，而是 MCP 的註冊作用範圍與 Session 載入狀態未符合目前專案。完成正確註冊並重新啟動 Session 後，原定工作流即可正常執行。
