# 繪圖規格（Drawing Spec）

本文件定義 flow-chart-creator skill 生成 draw.io 圖表時的共用品質要求，並以 User Flow 的自訂視覺與佈局規格為主。**在生成任何圖表之前，必須先完整讀過本規格**。

- User Flow：套用本文件的所有共用規則與 User Flow 自訂規則。
- Swimming Lane、ER Diagram、Sequence Diagram：套用本文件的共用品質要求，並依 draw.io MCP 提供的官方 XML 規則與該圖表的一般慣例生成；不強制套用 User Flow 的顏色分類或 U 型佈局。

## User Flow 顏色規範

繪製流程圖時，節點顏色應依下表統一配色，確保跨子流程的視覺一致性：

| 節點類型 | 顏色 | fillColor | strokeColor |
|----------|------|-----------|-------------|
| 使用者操作 | 藍色 | `#dae8fc` | `#6c8ebf` |
| 系統處理 | 綠色 | `#d5e8d4` | `#82b366` |
| MCP Server | 紫色 | `#e1d5e7` | `#9673a6` |
| 外部系統 | 橘色 | `#ffe6cc` | `#d79b00` |
| 決策節點 | 黃色 | `#fff2cc` | `#d6b656` |
| 開始／結束 | 灰色 | `#f5f5f5` | `#666666` |

## 共用品質要求

- 流程圖須使用 draw.io XML（mxGraphModel）生成，並透過 `open_drawio_xml` 開啟；不得使用 `open_drawio_mermaid`，也不得以 Mermaid 作為中間格式
- 產出的 XML 必須 well-formed（標籤正確閉合、屬性值跳脫 `<`/`>`/`&`/`"`、`mxCell` id 唯一），不可依賴 MCP server 端的後處理修補；上游已移除 XML normalization，任何不規範標記會直接導致 mxCodec 解析失敗或畫面空白
- User Flow 節點排列原則上由上到下（top-to-bottom），以符合一般閱讀習慣與流程邏輯順序
- 圖表需正確呈現所選圖表類型需要的角色、系統、資料實體與關係，不可遺漏重要元件
- 若流程涉及 MCP server、外部服務、後端系統或跨系統串接，須於圖中明確標示
- User Flow 中若涉及外部系統，應將外部系統（含 MCP Server）獨立繪製為系統方塊，並以帶標籤的箭頭標示資料傳送方向，例如「呼叫查詢設備清單」、「回傳設備清單」；不應以文字說明代替箭頭標籤
- User Flow 繪製外部系統互動時，建議採用「U 型資料流」佈局：主流程節點向右發出呼叫箭頭 → MCP Server → 外部系統 → 回傳箭頭往左進入下一個主流程節點，形成視覺上獨立的系統互動路徑
- 每張 User Flow 應在頂部附上顏色圖例（Legend），標示各顏色代表的節點類型
- User Flow 與 Swimming Lane 的判斷條件需以清楚的決策節點呈現，並完整標示成功、失敗、回圈或分支路徑
- **連線繞線交給 libavoid**：呼叫 `open_drawio_xml` 時必須帶 `routing: "libavoid"`。它會保留你擺好的節點位置，只把所有連線自動重算成繞開方塊的直角線——連線壓線、穿越節點的問題交由它處理，整體以清楚易讀為原則
- **不要手刻連線路徑**：因為 libavoid 會重算並覆蓋所有連線，請勿在連線上手動加 `<mxPoint>` 轉折點，也不要設 `exitX`/`exitY`/`entryX`/`entryY`——手刻的路由會被 libavoid 覆蓋而失效。每條連線只需宣告 `source` 與 `target`
- 回頭箭頭（從下方節點指回上方節點）同樣只宣告 source/target，由 libavoid 自動繞開中間節點、與主流程線分開呈現獨立路徑，不需再手動指定走左側邊緣
- 各流程節點、決策節點與系統元件之間應保留足夠間距，避免畫面過於擁擠
- 連線上若有 label 文字，請根據文字長度自動調整節點之間的間距，確保連線夠長、label 不會與節點或其他元素重疊擠壓
- 應避免元件重疊、節點重疊、文字遮擋或佈局錯位等影響閱讀的情況
- 可依角色、系統類型或流程階段加入顏色區分，以提升辨識度與可讀性
- 同一類型的節點應盡量維持一致的圖形樣式與視覺規則，避免同圖中風格混亂
- User Flow 與 Swimming Lane 若可拆分為多個子流程，需清楚呈現子流程之間的銜接關係，避免主流程斷裂
- 流程圖應以「可閱讀、可編輯、可延伸」為原則，不應為了塞進單一畫面而過度壓縮內容
- 若使用者針對流程圖提出版面、顏色、系統顯示、節點位置或連線方式的修正意見，AI 應依回饋持續調整直到符合需求
